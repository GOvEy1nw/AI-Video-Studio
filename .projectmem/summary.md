## Project purpose
AI Video Studio is a local-first desktop app for AI image, video, and audio generation. Forked from `deepbeepmeep/LTX-Desktop-WanGP`, it is being reshaped from an LTX-model-centric video editor into a Freepik/Higgsfield-style creative studio powered exclusively by WanGP. Free and community-focused, not commercial.

## Decisions
- Project identity: AI Video Studio, a local-first desktop app for AI image, video, and audio generation. Forked from deepbeepmeep/LTX-Desktop-WanGP (which itself is a fork of Lightricks/LTX-Desktop). The app is being reshaped from a LTX-model-centric video editor into a Freepik/Higgsfield-style creative studio powered exclusively by WanGP. Initial product focus: QuickGen image generation. Later: QuickGen video, audio/TTS, then Production workflows. Free/community-focused, not commercial.
- WanGP-only generation mandate: All normal product generation must route through WanGP. No cloud/API generation paths (LTX API, fal.ai/ZIT, Gemini) should be exposed as normal features. If a model cannot run through WanGP, it must not be exposed as a normal QuickGen model. Desired-but-unsupported models are tracked separately, not integrated via custom direct pipelines.
- Local-first product principle: The default product must not require cloud API keys. Users should be able to run the app locally, configure local runtime/models, and generate locally. Prompt/media should not be sent to external generation services for normal operation.
- Inheritance rule — preserve inherited systems by default: If LTX-Desktop-WanGP already has a working system for a feature (project management, gallery, generation cards, metadata, history, backend architecture, frontend routing, editor), do NOT redesign or rebuild it unless required by the explicit goals of the PRD. Prefer extension over replacement. The inherited implementation remains the source of truth.
- Remove, disable, or hide inherited features that depend on external generation providers: LTX API-only generation paths, cloud generation provider flows, API key onboarding, fal.ai / Z-Image API usage, Gemini prompt suggestion API usage, any feature that sends prompts/media to external services, direct non-WanGP model-loading paths, cloud text encoding, API-only macOS assumptions, telemetry (unless fully removed or strictly opt-in and transparent), billing/cost/API quota copy, LTX Desktop branding/copy.
- Curated model profiles: Do not expose every WanGP-supported model automatically. The main UI should show a curated set of tested model profiles. A model profile answers: display name, media type, WanGP model/type/settings mapping, supported inputs, LoRA support status, resolution availability, aspect ratio availability, install/availability status, experimental/stable status. Add a lightweight curated model profile layer only if the inherited app does not provide an equivalent clean mechanism.
- QuickGen-first development order: Images first, then Video, then Audio/TTS, and finally Production. QuickGen should be the fast creative loop (Prompt → Generate → Compare → Reuse → Vary → Save). QuickGen controls: media type selector, model selector, aspect ratio, resolution, prompt, reference input where supported, LoRA selector where supported, seed lock, generate/cancel/progress. Advanced settings may exist but should not dominate the main experience.
- Production tab deferred: Production (structured workflow for finished AI video projects) should be visible but disabled or marked as coming later — not built before QuickGen is stable. It should eventually support: idea/treatment, script, visual style, characters, locations, props, shot list, storyboards, per-shot prompts, image/video/audio generation, review/approval, export/editor handoff.
- LoRA support is a key differentiator for the local-first advantage. MVP scope: LoRA folder detection/configuration, compatible LoRA listing, manual LoRA import/copy, enable/disable per generation, strength slider where WanGP supports it, LoRA info saved with generation metadata. LoRA should appear as an optional QuickGen control (e.g. 'LoRA: None / Select...'), hidden/disabled for models where LoRA is not available.
- Video editor is a bonus, not the MVP centre. Keep the inherited editor code if it does not slow development. Hide it, beta-label it, or leave it as-is. Do not rewrite it for MVP. Future use: review area for generated clips, simple assembly space, variation comparison, bridge from QuickGen into Production, rough AI animatic tool.
- MVP 0.1 — Local-Only QuickGen Image Foundation: Fork LTX-Desktop-WanGP into AI Video Studio. Keep inherited project/generation/gallery foundations. Remove/disable visible cloud/API usage. Generate images locally through WanGP only. Support at least one stable WanGP image model end-to-end. Introduce simple curated WanGP model addition. Keep output/history aligned with inherited app. Keep Production visible but disabled or marked as coming later.

Key findings: (1) 11 distinct generation routes identified — 4 WanGP (keep), 3 cloud/API (remove), 4 local non-WanGP (remove after WanGP stable). ## Notes
- Cloud/API dependencies in inherited codebase that need removal/disabling: (1) LTXAPIClient — used for cloud video generation, text encoding, retake; (2) ZitAPIClient — fal.ai Z-Image Turbo image generation; (3) Gemini prompt suggestion API in SuggestGapPromptHandler; (4) LTXTextEncoder with cloud text encoding support; (5) LTXRetakePipeline; (6) ZitImageGenerationPipeline; (7) LTXFastVideoPipeline for non-WanGP local video gen; (8) API key settings in SettingsModal and settings.json (ltx_api_key, fal_api_key); (9) analytics.ts telemetry; (10) billing/cost copy in UI.
- Inherited feature inventory (preserve by default): (1) Project system — ProjectContext with create/open/delete/rename projects, assets (Asset/AssetTake), timelines, favorites, view routing (home/project/playground), project persistence via electron-store; (2) Gallery/Gen Space — GenSpace.tsx with generation cards, prompt input, model selection, aspect ratio/resolution controls, seed, progress/cancel, output display; (3) Generation history — metadata stored with each generation; (4) Video editor — VideoEditor.tsx with timeline, tracks, clips, subtitles, gap fill generation, retake; (5) Settings — SettingsModal.tsx for API keys, model settings, text encoding; (6) Home view with project list and playback menu.
- User-facing views in inherited frontend: Home.tsx (project list, playground, startup), GenSpace.tsx (QuickGen/generation space), VideoEditor.tsx (full video editor with timeline), SettingsModal.tsx (app settings). View routing via ProjectContext currentView: 'home', 'project', 'playground'. GenSpace is the primary target for QuickGen adaptation — already has prompt, model selection, resolution, reference input, generation/cancel/progress, and output gallery cards.
- Branding/rebranding targets: electron-builder.yml has productName 'LTX Desktop', appId 'com.lightricks.ltx-desktop', copyright Lightricks, code signing with Lightricks Ltd, shortcutName 'LTX Desktop', publish target Lightricks/ltx-desktop. package.json has name 'ltx-desktop', description 'LTX-2 Video Generation - Desktop App', author Lightricks. Frontend likely has 'LTX Desktop' in window titles, splash screen, about dialog, and UI copy. App data folder is %LOCALAPPDATA%/LTXDesktop/ on Windows.
- Runtime config: Python backend managed by Electron main process (electron/python-backend.ts). RuntimeConfig built from env vars + settings. WanGP detection: local ./Wan2GP checkout preferred, then WANGP_ROOT env var. Backend venv isolated with PYTHONNOUSERSITE=1. setup:dev:win clones Wan2GP, installs backend deps. setup:dev:linux builds for external checkout. dev starts Vite + Python backend concurrently. dev:debug adds Electron inspector + debugpy.
- Phase 0 (current): Fork audit and preservation map. Create clear map of what must be kept, removed, extended, and left untouched. Identify all generation routes, cloud/API paths, WanGP paths, runtime setup assumptions. Document inherited project/generation/gallery/card/metadata/job behaviour. Phase 1: Local-only product shell — rename/rebrand to AI Video Studio, remove/hide API key onboarding, remove/hide cloud provider settings, remove/disable telemetry, update copy to local/WanGP positioning, add WanGP attribution.
- Hard guardrails for AI coding agents working on this project: (1) Do not rebuild the project system; (2) Do not rebuild gallery/generation cards; (3) Do not redesign metadata storage unless it cannot support WanGP/model/LoRA fields; (4) Do not replace Electron/React/FastAPI architecture; (5) Do not add generation bypassing WanGP; (6) Do not expose cloud/API generation as normal; (7) Do not require API keys for MVP; (8) Do not add custom model pipelines for models WanGP doesn't support; (9) Do not expose every WanGP setting in QuickGen; (10) Do not expose every WanGP model automatically; (11) Do not build Production before QuickGen stable; (12) Do not let video editor define MVP; (13) Do not hide WanGP attribution.
- WanGP bridge (backend/services/wangp_bridge.py): The bridge delegates /api/generate and /api/generate-image to an existing WanGP installation. Uses WanGP's shared.api.WanGPSession for in-process execution. Video requests translated into single-task WanGP manifests. Image requests use the same mechanism with the configured image model. WANGP_ROOT env var points to WanGP checkout (folder containing wgp.py). Optional config: WANGP_VIDEO_MODEL_TYPE (default ltx2_22B_distilled), WANGP_IMAGE_MODEL_TYPE (default z_image), WANGP_EXTRA_ARGS.
- Project purpose in PROJECT_MAP.md has been populated with the full AI Video Studio description per AGENTS_PRD.md. This note triggers regeneration of summary.md to pick up the new Project purpose section from the map.
- WanGP pip dependencies can go missing from the backend venv (e.g. mmgp==3.7.9). When this happens, WanGP's wgp.py fails with ModuleNotFoundError: No module named 'mmgp', but the generation routing is correct — it passes through _generate_via_wangp → WanGPBridge → session.submit_manifest(). The fix is to re-run pnpm setup:dev:win, which reinstalls all Wan2GP requirements.txt into the backend venv.
